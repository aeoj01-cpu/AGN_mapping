### Debugging Accretion Disk Convolution Normalisation

While testing the accretion disk response via convolution with the driving light curve, I encountered an issue where the convolved (disk) light curve had a significantly larger amplitude than the original driving signal. This was initially unexpected, as a properly normalised transfer function should produce a smoothed and delayed signal with a comparable (or smaller) amplitude.

The issue was traced to a mismatch between **continuous** and **discrete** normalisation of the transfer function. I had originally normalised the disk response function using numerical integration:

```
normi = np.trapz(combi_psi, x=times)
psi_norm = combi_psi / normi
```

This enforces:  
∫ψ(t) dt = 1

However, the convolution is performed numerically as a **discrete sum**, not a continuous integral:

F_disk[i] = Σ F[i−j] · ψ[j]

In this formulation, the correct normalisation condition is:

Σψ[j] = 1

By normalising with the integral, I effectively imposed:

Σψ[j] ≈ 1/Δt

which led to an artificial amplification of the convolved signal by a factor ≈ 1/Δt (~50 for a 30-minute cadence). This explains why the disk light curve amplitude was significantly larger than the driving light curve.

The fix was to normalise the transfer function using a discrete sum instead:

```
psi_norm = combi_psi / np.sum(combi_psi)
```

After applying this correction, the convolved disk light curve behaved as expected: it is now a smoothed, delayed version of the driving light curve with a physically reasonable amplitude.

This highlights the importance of ensuring consistency between continuous theoretical formulations and their discrete numerical implementations, particularly when working with convolution in time-domain simulations.
